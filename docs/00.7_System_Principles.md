Purpose

This document defines the engineering and architectural principles that govern the entire Dhanashree ERP platform.

These principles are mandatory.

Every feature, database table, API endpoint, background job, AI workflow, and frontend screen must comply with these rules.

Whenever a design decision is unclear, this document takes precedence over implementation convenience.

⸻

Engineering Philosophy

Dhanashree ERP is not built around screens.

It is built around business operations.

The software exists to model how a contractor actually works.

Technology serves the business—not the other way around.

⸻

Core Principle 1

The Project is the Center of the System

Every operational activity should belong to a project.

Examples

* Material Purchase
* Labour Attendance
* Supplier Invoice
* Client Payment
* Miscellaneous Expense
* Site Progress
* Documents

Only company-level expenses are allowed to exist outside projects.

⸻

Core Principle 2

Every Rupee Must Be Explainable

Every financial number shown anywhere in the application must be traceable.

The owner must always be able to answer:

* Where did this money come from?
* Where did this money go?
* Which project?
* Which supplier?
* Which document proves it?
* Who entered it?
* When was it entered?

No unexplained financial values may exist.

⸻

Core Principle 3

Source Data is Sacred

Only original business events are stored.

Everything else is calculated.

Examples

Store

* Transaction
* Attendance
* Payment
* Invoice
* Work Order

Never Store

* Profit
* Remaining Balance
* Total Cost
* Salary Due
* Cashflow
* Project Margin

Derived values are computed from source data.

⸻

Core Principle 4

Transactions are Immutable

Transactions should never be edited or deleted after they become part of the financial history.

Corrections must occur using:

* Reversal Entries
* Adjustment Transactions
* Audit Records

This preserves financial integrity.

⸻

Core Principle 5

Every Action Creates History

Every important action creates an audit record.

Examples

Project Created

Transaction Added

Attendance Updated

Supplier Paid

Invoice Uploaded

Salary Released

Audit history is permanent.

⸻

Core Principle 6

Documents are Evidence

Documents support business activities.

Documents do not replace business data.

Example

Invoice

↓

OCR

↓

Transaction

↓

Supplier Ledger

↓

Cashflow

The document remains attached as supporting evidence.

⸻

Core Principle 7

AI Assists — AI Never Decides

Artificial Intelligence may

* Suggest
* Predict
* Categorize
* Extract
* Recommend

Artificial Intelligence must never

* Delete Data
* Approve Payments
* Change Financial Records
* Close Projects
* Release Salaries

Final decisions always belong to a human.

⸻

Core Principle 8

Mobile First

The owner should comfortably operate the software on a phone.

Desktop layouts are enhancements.

Every important action should require as few interactions as possible.

⸻

Core Principle 9

Minimize Manual Entry

The software should eliminate repetitive typing.

Preferred methods

* OCR
* Smart Defaults
* Auto Complete
* Templates
* Recently Used Values

Typing should always be the last option.

⸻

Data Principles

Single Source of Truth

Every business concept has exactly one authoritative location.

Examples

Projects Table

Owns projects.

Transactions Table

Owns financial records.

Attendance Table

Owns attendance.

No duplicate sources are allowed.

⸻

Referential Integrity

Every relationship must remain valid.

Examples

A Transaction cannot reference a missing Project.

A Payment cannot reference a missing Transaction.

A Labour Attendance cannot reference a missing Labour.

⸻

Soft Deletes

Business data should never be permanently removed.

Instead

* Archived
* Inactive
* Cancelled

This preserves history and reporting accuracy.

⸻

Version History

Important documents should maintain revisions.

Examples

Quotation

Version 1

↓

Version 2

↓

Version 3

Work Orders may also maintain revisions if reissued.

⸻

Financial Principles

Every financial event must satisfy these rules.

Rule 1

Every expense belongs to a category.

⸻

Rule 2

Every expense belongs to a project unless explicitly marked as a company expense.

⸻

Rule 3

Every payment settles one or more transactions.

⸻

Rule 4

Cashflow is calculated.

Never stored.

⸻

Rule 5

Profit is calculated.

Never stored.

⸻

Rule 6

Outstanding balances are calculated.

Never stored.

⸻

Rule 7

Supplier Ledgers are generated from transactions.

Never maintained manually.

⸻

User Experience Principles

The application should answer important questions within ten seconds.

Examples

* Which projects are profitable?
* Which supplier needs payment?
* Which labour salary is pending?
* What did we spend today?
* Which invoices remain unpaid?

Navigation should remain simple enough that a first-time user can understand the application without formal training.

⸻

AI Principles

AI should improve speed—not reduce trust.

Acceptable AI tasks

* OCR
* Invoice Parsing
* Categorization
* Search
* Recommendations
* Duplicate Detection
* Smart Suggestions

AI outputs must always be reviewable before becoming permanent.

⸻

Security Principles

Every authenticated action should be attributable to a user.

Sensitive operations require elevated permissions.

Financial records should be tamper-resistant.

Passwords must never be stored directly.

Files should be securely stored with controlled access.

⸻

Development Principles

Keep modules loosely coupled.

Prefer composition over duplication.

Business logic belongs in backend services.

Frontend should remain presentation-focused.

Naming should follow business terminology defined in the Business Domain document.

Every API should be predictable and RESTful.

⸻

Scalability Principles

The architecture should support future expansion without redesign.

Future capabilities include

* Multiple Companies
* Multiple Owners
* Multiple Teams
* Multiple Regions
* Inventory
* Warehouses
* Fleet Management
* Purchase Orders
* Client Portal
* Vendor Portal
* Mobile Applications

Current implementation decisions should not prevent these future additions.

⸻

Non-Negotiable Rules

These rules apply to every contributor, human or AI.

1. Never duplicate business logic.
2. Never store calculated values.
3. Never bypass audit logging.
4. Never delete financial history.
5. Never trust client-side calculations.
6. Never allow orphaned records.
7. Every financial change must be traceable.
8. Every document must remain linked to its business context.
9. AI suggestions require user confirmation.
10. Business terminology must remain consistent across the platform.

⸻

Final Principle

Every feature should answer this question:

Does this make the owner’s business easier to understand and operate?

If the answer is no, the feature should be reconsidered.

⸻

Conclusion

The System Principles define the architectural DNA of Dhanashree ERP.

These principles ensure that the platform remains trustworthy, maintainable, scalable, and financially accurate as it grows from a single-company solution into a full contractor operating system.

Every future document and every line of code must align with the principles established here.

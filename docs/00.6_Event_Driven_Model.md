Purpose

This document defines how every business activity propagates throughout the Dhanashree ERP system.

Rather than treating the application as isolated CRUD operations, the system is designed around business events.

Whenever something important happens inside the business, an event is emitted.

Other parts of the application react to these events automatically.

This architecture minimizes duplicate logic while ensuring every module remains synchronized.

⸻

Event Driven Philosophy

The software does not react to screens.

It reacts to business events.

Example

Instead of:

User clicks Save.

The system thinks:

“Material Purchased”

Everything else reacts.

Event Rules

Rule 1

Every business event must be timestamped.

⸻

Rule 2

Events are immutable.

Events are never edited.

New events replace old state.

⸻

Rule 3

One business action creates one primary event.

That event may trigger multiple secondary events.

⸻

Rule 4

Events should never directly manipulate unrelated modules.

Communication happens through event handlers.

⸻

Rule 5

If an event fails during processing, the system must rollback partial updates where transactional consistency is required, or record compensating actions for asynchronous processes.

Event Philosophy

Business events represent facts.

Examples

Material Purchased

Salary Paid

Client Paid Advance

Invoice Uploaded

These facts never change.

The system state is derived from these facts.

⸻

Conclusion

The Event Driven Model ensures every module inside Dhanashree ERP remains synchronized without duplicating business logic.

Instead of tightly coupling features together, every module reacts to well-defined business events.

This architecture provides scalability, maintainability, auditability, and a strong foundation for realtime updates, AI workflows, OCR processing, notifications, and future automation.
